

気のせいではない。発動が悪い原因は中身ではなく、ほぼ確実に description です。

問題の核心は frontmatter のこれ:

```
description: LLMくさい日本語、翻訳調・硬すぎる日本語を、原意と技術的意味を保ったまま自然で読みやすい日本語に修正する。
```

スキルの起動判定はこの description 一行でしか行われません。本文がいくら充実していても、起動前のモデルは本文を見ていない。なのに今の description は「何をするスキルか」の説明にはなっているが、「いつ起動すべきか(トリガー)」の手がかりが薄い。

効きを良くするための具体的な直し:

1. トリガー語を明示的に並べる。「README」「仕様書」「技術記事」「コメント」「エラーメッセージ」「issue」「PR」「職務経歴書」「Zenn記事」など、対象になる名詞を description に直接入れる。あなたの本文には載っているのに description には一つも入っていない。

2. ユーザーが実際に発する言い回しを含める。「日本語を自然にして」「翻訳調を直して」「LLMっぽい文を直して」「読みやすくして」「校正して」「リライトして」など、依頼の表層形。description は本文と違って、ユーザーの発話とのマッチングに使われる。

3. 三人称・トリガー型の書き方にする。Anthropic公式スタイルは「Use when... / Use this skill whenever the user wants to...」型。日本語スキルでも「〜したいとき」「〜を含む依頼で起動」のように、条件記述にすると判定が安定します。

書き換え案:

```
description: >-
  日本語テキストを自然で読みやすく修正・リライト・校正したいときに使う。
  LLMくさい日本語、翻訳調、硬すぎる日本語、文意の取りづらい悪文が対象。
  README、仕様書、技術記事、Zenn記事、コードコメント、docstring、
  エラーメッセージ、UI文言、issue、PR説明文、職務経歴書などの日本語を、
  原意・事実・仕様・技術的意味を保ったまま整える。「自然にして」「翻訳調を直して」
  「読みやすくして」「LLMっぽさを消して」「リライトして」「校正して」などの依頼で起動する。
```

もう一点、起動とは別に効果が薄く見える原因として考えられるのが、本文が38KB・1422行と長大なことです。起動はしても、モデルが全ルールを均等に適用しきれず「効いてる感」が薄れる。Step群・修正観点・典型例11個が重複気味なので、本文を「コア原則(述語中心の項/付加部分析)+ チェックリスト」に圧縮し、典型例や細則は別ファイル(`reference.md` 等)に切り出して必要時のみ読ませる構成にすると、適用が安定します。

どちらが主因か切り分けるなら、まず「起動するか」を確認するのが先です。明示的に「natural-japanese-rewriter スキルを使って」と指定したときはちゃんと効くか、それとも指定しても効きが悪いか、どちらですか。前者なら description、後者なら本文の問題です。


分割するなら、各スキルが「単一の判断軸」を持つように切るのが基本です。今の SKILL.md は実は工程がすでに Step 1〜10 で直列化されているので、それを自然な切れ目で束ねると以下のようになります。

直列パイプライン案(4段)

```
[1] 保護・対象判定     → 触っていい箇所/触ってはいけない箇所を確定
[2] 構造分析(述語中心) → 述語・項・付加部・骨格を抽出、欠落と係り先を判定
[3] リライト実行       → 構造に基づき語順・分割・助詞・冗長表現を修正
[4] 検証・要確認出力   → 意味保持チェックリスト、判断不能箇所を報告
```

各段の責務:

1. **scope-guard**(Step 1〜3相当): コード・識別子・URL・コードブロック・自動生成ファイルを保護対象として確定し、文体(です/ます・である)を判定。出力は「編集可能な範囲のテキスト + 保護リスト + 文体タグ」。ここは判断のみで本文を書き換えない。

2. **predicate-analyzer**(Step 4〜7相当): 各文を述語ごとに分解し、項/付加部、骨格、欠落項(復元可/不可)、係り先の遠さをマーキングする。出力は本文ではなく**注釈付き中間表現**(下記)。このスキルの肝は「ここでは直さない」こと。分析と修正を混ぜると、今の効きの悪さが再発します。

3. **rewriter**(Step 8〜9相当): 中間表現を入力に、語順整理・文分割・助詞修正・LLMくさい表現の除去を実行。意味を変える補完は禁止のまま。

4. **verifier**(Step 10 + 最終チェックリスト相当): 否定/条件/可能性/主体などが保たれているか検査し、「要確認」を集約して最終出力形式に整える。

段と段の受け渡し(ここが直列化の成否を分ける)

直列で走らせるなら、各段の**出力スキーマを固定**する必要があります。自然文で渡すと次の段が再解析してブレる。たとえば中間表現を JSONL にする:

```jsonl
{"id": 1, "text": "ユーザーが入力した値を検証し、問題がなければ保存します。",
 "predicates": [
   {"verb": "検証する", "args": {"が": "(省略)", "を": "入力した値"}, "adjuncts": []},
   {"verb": "保存する", "args": {"が": "(省略・復元可)", "を": "(省略・復元可)"}, "adjuncts": [{"type": "条件", "text": "問題がなければ"}]}
 ],
 "issues": [],
 "protected": []}
```

rewriter はこれだけを見て直し、verifier は元 text と最終 text を突き合わせる。

実装上の現実的な注意

- **オーケストレーションの主体**: Claude Code/Amp のスキルは LLM が「呼ぶ」もので、シェルパイプのように自動直列実行はされません。だから実体は「親スキル(orchestrator)が子スキルを順に参照する」形になる。`natural-japanese-rewriter/SKILL.md` を薄いオーケストレータにして、`scope-guard.md` `predicate-analyzer.md` … を `reference` として順に読ませるのが一番素直です。

- **分割しすぎの罠**: 4段でも、毎段でコンテキストに本文+前段出力が積み上がる。短文の校正なら overhead が利得を上回ります。**分割が効くのは中〜長文・複数文書のとき**。短い UI 文言は単段で済ませる分岐を orchestrator に持たせるとよい。

- **2と3の分離が最重要**: 今の効きの悪さは、分析と修正が同じプロンプト内で同時進行して、モデルが「分析した気になって直す」ために起きている可能性が高い。predicate-analyzer を独立段にして**中間表現を必ず出力させる**だけで、体感はかなり変わるはずです。

最小構成にするなら、まず2段(analyze → rewrite+verify)で試して、効果を見てから verifier を切り出すのが現実的です。全部いきなり4段にすると、受け渡しスキーマの設計コストが先に来て頓挫しやすい。

どの粒度の文書が主対象ですか。職務経歴書中心なら verifier を厚く、技術記事中心なら predicate-analyzer を厚く、という重み付けが変わります。
