# natural-japanese-rewriter 改修実行計画

## 1. 要件サマリ

`skills/natural-japanese-rewriter` の発動率と実行品質を改善する。
参考メモ `docs/2026_06_25.md` の指摘どおり、まず frontmatter の `description` を実際の依頼文に寄せて改善する。
そのうえで、1422 行ある単一 `SKILL.md` を、発動・オーケストレーション・最終リライトを担う既存スキルと、前段の文意分析を担う新規スキルの二つに分割し、既存スキルから新規スキルを先に使う直列実行手順を明文化する。
改修前には既存の `skills/natural-japanese-rewriter/` をバックアップし、分割後でも元の内容を参照・復元できるようにする。

## 2. 前提条件

- 現在の `skills/natural-japanese-rewriter/SKILL.md` は 1422 行あり、述語・項・付加部・骨格分析からリライト例までを単一ファイルに含んでいる。
- 現在の `description` は「LLMくさい日本語、翻訳調・硬すぎる日本語...」に寄っており、README、仕様書、PR説明文、UI文言など、ユーザーが依頼時に使う対象名詞や表層表現が不足している。
- スキルの起動判定は、起動前に本文を読めないため、frontmatter の `description` に強く依存する。
- Codex/Amp のスキルはシェルパイプのように自動連結されるものではない。直列実行は、親側の `SKILL.md` に「最初に分析スキルを読み、その出力契約に基づいてリライトする」と明記して実現する。
- 既存の `agents/openai.yaml` と `manifest.json` も、分割後のファイル構成に合わせて更新が必要になる。

## 3. 制約

- 原意、事実、仕様、技術的意味を変えないことを最優先にする。
- 発動率を上げるために description を広げるが、単なる翻訳、創作、要約、内容変更の依頼まで過剰に拾わない。
- 二分割を優先し、4 段以上の細分化は今回の実装範囲に入れない。参考メモの 4 段案は、二分割内の責務整理として吸収する。
- 短い UI 文言や 1 文だけの軽微な修正では、分析結果をユーザーに冗長に見せず、内部手順として扱う。
- README のインストール手順に新規スキルを追加する場合、既存のコピー・シンボリックリンク手順と形式を揃える。
- バックアップは `skills/` 配下に置かない。別スキルとして誤認されないように、`docs/backups/2026_06_25/natural-japanese-rewriter/` に退避する。

## 4. 優先順位

1. スキル発動率の改善: `description` に対象文書名と依頼表現を明示する。
2. 意味保持の安全性: 分析スキルで保護範囲、述語、項、付加部、曖昧さを先に確定する。
3. 直列実行の安定性: 既存スキルから分析スキルへ進み、その分析結果をリライト入力として使う契約を固定する。
4. 本文の軽量化: 既存の重複ルールと例を整理し、必要なら `references/` に退避する。
5. 既存利用者への影響最小化: `natural-japanese-rewriter` は入口として残す。

## 5. 分割方針

### 採用案: 既存スキルを入口に残し、分析スキルを新設する

| 項目 | 内容 |
| --- | --- |
| 既存スキル | `skills/natural-japanese-rewriter` |
| 新規スキル | `skills/natural-japanese-rewrite-analyzer` |
| 実行順 | `natural-japanese-rewriter` -> `natural-japanese-rewrite-analyzer` -> `natural-japanese-rewriter` 内のリライト・検証 |
| 狙い | ユーザーの通常依頼は既存名で拾い、内部手順として分析を先に走らせる |
| トレードオフ | 完全に独立した二つのスキルではなく、既存スキルが親スキル兼リライト担当になる |

不採用案は、`analyzer` と `applier` の二つを完全に独立させる構成。
この場合、通常の「日本語を自然にして」という依頼で両方が必ず選ばれる保証が弱いため、今回は採用しない。

## 6. 予定ファイル構成

```text
skills/
  natural-japanese-rewriter/
    SKILL.md
    manifest.json
    agents/openai.yaml
    references/
      examples.md
      detailed-rules.md
  natural-japanese-rewrite-analyzer/
    SKILL.md
    manifest.json
    agents/openai.yaml
```

`references/` は必須ではないが、既存 `SKILL.md` から典型例や細則を退避して本文を軽くする場合に使う。
最初の実装では、本文の大幅削除よりも、分割と直列実行契約の確立を優先する。

## 7. description 改善案

### `natural-japanese-rewriter`

```yaml
description: >-
  日本語テキストを自然で読みやすく修正・リライト・校正したいときに使う。
  LLMくさい日本語、翻訳調、硬すぎる日本語、文意の取りづらい悪文が対象。
  README、仕様書、技術記事、Zenn記事、コードコメント、docstring、
  エラーメッセージ、UI文言、issue、PR説明文、職務経歴書などの日本語を、
  原意・事実・仕様・技術的意味を保ったまま整える。
  「自然にして」「翻訳調を直して」「読みやすくして」「LLMっぽさを消して」
  「リライトして」「校正して」などの依頼で起動する。
```

### `natural-japanese-rewrite-analyzer`

```yaml
description: >-
  日本語のリライト前に、原文の保護範囲、述語、項、付加部、文の骨格、
  曖昧な箇所を分析したいときに使う。
  natural-japanese-rewriter の前段として、原意・事実・仕様・技術的意味を変えずに
  書き換えるための分析メモを作る。
```

## 8. 直列実行の契約

`natural-japanese-rewriter/SKILL.md` の冒頭に、次の手順を明記する。

1. 入力が日本語リライト対象か判定する。
2. コードブロック、コマンド、構造化データ、固有名詞、仕様値などの保護範囲を確認する。
3. `../natural-japanese-rewrite-analyzer/SKILL.md` を読み、分析手順を適用する。
4. 分析結果を次の形式で内部メモ化する。

```yaml
protected_ranges:
  - 変更してはいけない範囲
rewrite_targets:
  - 不自然さ、翻訳調、冗長さ、係り受けの問題
predicates:
  - 述語と対応する項
adjuncts:
  - 条件、理由、目的、時点などの付加部
ambiguities:
  - 原文だけでは判断できない箇所
rewrite_policy:
  - 分割、語順変更、助詞調整、省略維持、明示化の方針
```

5. 分析結果に基づいてリライトする。
6. 原意、事実、仕様、技術的意味が変わっていないか検証する。
7. 判断不能な箇所がある場合だけ、`要確認` として短く報告する。

## 9. 実装手順

1. 既存スキルのバックアップ作成
   - `skills/natural-japanese-rewriter/` を `docs/backups/2026_06_25/natural-japanese-rewriter/` に丸ごとコピーする。
   - バックアップ対象は `SKILL.md`、`manifest.json`、`agents/openai.yaml` を含む既存ディレクトリ全体とする。
   - バックアップは復元用というより、分割作業中に元の表現、例、細則を参照するための比較元として扱う。
   - `docs/backups/` 配下はスキルのインストール対象に含めない。

2. ベースライン確認
   - `skills/natural-japanese-rewriter/SKILL.md` の現行 frontmatter、見出し構成、重複している例示ブロックを確認する。
   - `README.md` のインストール手順に新規スキル追加が必要か確認する。

3. `natural-japanese-rewriter` の description 更新
   - 参考メモの改善案をベースに、対象文書名と依頼表現を frontmatter に追加する。
   - `agents/openai.yaml` の `short_description` も必要に応じて同じ方向で短く更新する。

4. 分析スキルの新設
   - `skills/natural-japanese-rewrite-analyzer/` を作成する。
   - `SKILL.md` には、保護範囲判定、述語、項、付加部、骨格、曖昧さ検出、分析メモ形式だけを置く。
   - `manifest.json` と `agents/openai.yaml` を追加する。

5. 既存スキルの親スキル化
   - 既存 `SKILL.md` の冒頭に、分析スキルを先に適用する直列手順を追加する。
   - 既存本文から、分析スキルへ移すルールと、リライト側に残すルールを分ける。
   - リライト側には、語順、文分割、助詞、冗長表現、自然な省略、検証、要確認出力を残す。

6. 長大な例・細則の退避
   - 既存本文の典型例や細かい判断基準が重複している場合、`references/examples.md` または `references/detailed-rules.md` に移す。
   - `SKILL.md` には「必要な場合だけ references を読む」と書き、常時読む本文を短くする。

7. ドキュメント更新
   - `README.md` のコピー手順とシンボリックリンク手順に `natural-japanese-rewrite-analyzer` を追加する。
   - 分割後の使い方として、通常は `natural-japanese-rewriter` を入口にすることを明記する。

8. 検証
   - `name:` と `description:` の frontmatter が妥当な YAML として読めることを確認する。
   - `manifest.json` の `files` に実在ファイルだけが入っていることを確認する。
   - `docs/backups/2026_06_25/natural-japanese-rewriter/` に改修前の既存スキル一式が残っていることを確認する。
   - 新規スキルを `$CODEX_HOME/skills` にシンボリックリンクして、新しい Codex セッションでスキル一覧に出ることを確認する。
   - 次の依頼文で `natural-japanese-rewriter` が起動しやすいか確認する。
     - 「この README の日本語を自然にして」
     - 「PR 説明文の LLM っぽさを消して」
     - 「この UI 文言を読みやすく校正して」
     - 「職務経歴書の翻訳調を直して」
   - 長めの技術文で、分析、リライト、検証の順に出力品質が落ちていないか確認する。

## 10. リスクと対策

| リスク | 対策 |
| --- | --- |
| description を広げすぎて無関係な依頼で起動する | 「日本語テキストの修正・リライト・校正」に用途を限定する |
| 分割作業で既存の長いルールや例を失う | 作業前に `docs/backups/2026_06_25/natural-japanese-rewriter/` に既存スキルを丸ごと保存する |
| 分析スキルだけが単独起動して最終リライトされない | 通常入口は `natural-japanese-rewriter` と README に明記する |
| 二分割後も本文が長く、効果が薄い | examples と detailed-rules を references に退避する |
| 直列実行がモデル依存で揺れる | `SKILL.md` 冒頭に実行順と分析メモ形式を固定する |
| 意味保持より自然さを優先しすぎる | 検証手順に原意・事実・仕様・技術的意味のチェックを必須化する |
