# Natural Japanese Rewriter

日本語テキストを、原意・事実・仕様・技術的意味を保ったまま、自然で読みやすい日本語に整える。
対象は README、仕様書、技術記事、Zenn 記事、コードコメント、docstring、エラーメッセージ、UI 文言、issue、PR 説明文、職務経歴書など。

## 最優先ルール

- 原意を変えない。
- 事実を変えない。
- 仕様を変えない。
- 技術的意味を変えない。
- 原文から判断できない情報を補わない。
- コード、コマンド、ファイルパス、API 名、設定値、数値、固有名詞、引用、構造化データは、明確な誤りがない限り保持する。

自然さより意味保持を優先する。
自然にするために意味を足す必要がある場合は、本文へ混ぜずに `要確認` として短く分ける。

## 直列実行

このスキルを使うときは、原則としてリライト前に `natural-japanese-rewrite-analyzer` を前段として使う。
同じ `skills/` ディレクトリに `natural-japanese-rewrite-analyzer` がある場合は、その `SKILL.md` を読み、分析手順を適用してから書き換える。
利用できない場合は、このスキル内の「分析メモ形式」に沿って同等の分析を内部で行う。

短い UI 文言や 1 文だけの軽微な修正では、分析結果をユーザーへ冗長に出さない。
中長文、仕様文、技術文、職務経歴書、意味の取りづらい文では、分析結果を内部メモとして必ず作ってからリライトする。

## 分析メモ形式

リライト前に、次の形式で内部メモを作る。
通常はユーザーに出力しない。

```yaml
protected_ranges:
  - 変更してはいけない語句、コード、数値、仕様、固有名詞、引用
rewrite_targets:
  - 不自然さ、翻訳調、冗長さ、係り受け、助詞、文分割の問題
predicates:
  - 述語と対応する項
adjuncts:
  - 条件、理由、目的、時点、範囲などの付加部
ambiguities:
  - 原文だけでは判断できない箇所
rewrite_policy:
  - 分割、語順変更、助詞調整、省略維持、明示化の方針
```

## リライト手順

1. 入力が日本語リライト対象か確認する。
2. 触ってよい範囲と保護すべき範囲を分ける。
3. `natural-japanese-rewrite-analyzer` の分析手順で、述語、項、付加部、文の骨格、曖昧さを確認する。
4. 分析メモに基づいて、語順、文分割、助詞、冗長表現、表現の硬さを整える。
5. 省略された主語や目的語は、文脈から一意に復元できる場合だけ補う。
6. 補うとかえって不自然な項は、日本語として自然な省略のまま残す。
7. リライト後に、原意・事実・仕様・技術的意味が変わっていないか確認する。
8. 判断不能な箇所がある場合だけ、`要確認` として短く報告する。

## 書き換え方針

### 語順

日本語では、述語の近くに重要な項を置くと読みやすくなる。
長い条件、理由、目的、補足説明が文の骨格を圧迫している場合は、文を分ける。
ただし、文を分けることで対比、因果、条件が失われる場合は分けない。

### 助詞

助詞は意味関係を変えない範囲で自然に整える。
`は` と `が`、`を` と `に`、`で` と `により` などを入れ替える場合は、主題、焦点、対象、手段、場所、原因が変わらないか確認する。

### 冗長表現

LLM らしい定型句、翻訳調の直訳、過度に硬い名詞句は簡潔にする。
ただし、仕様書、契約文、職務経歴書などで必要な厳密さは残す。

### 文分割

一文に複数の条件、理由、例外、結果が詰まっている場合は分割する。
Markdown の箇条書き、表、コードブロック、JSON / YAML / TOML、コマンド例は構造を壊さない。

### 曖昧さ

原文だけでは判断できない曖昧さは、勝手に決めない。
曖昧さを残すと読みにくい場合は、自然な範囲でぼかすか、`要確認` に分ける。

## 詳細ルール

通常はこの `SKILL.md` だけで実行する。
長文、技術的な曖昧さが多い文、判断に迷う文、既存ルールとの照合が必要な文では、`references/detailed-rules.md` を読む。
この参照ファイルには、改修前の詳細なルールと例を保存している。

## 出力

ユーザーが形式を指定していない場合は、修正後の本文を先に出す。
必要な場合だけ、短い補足として主な修正点や `要確認` を続ける。
過剰な解説、分析メモ全体、内部手順の説明は出さない。
