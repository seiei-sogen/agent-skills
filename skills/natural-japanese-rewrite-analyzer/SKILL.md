# Natural Japanese Rewrite Analyzer

日本語を自然に書き換える前に、原文の意味構造を確認するための分析スキル。
最終的なリライトは `natural-japanese-rewriter` が担当し、このスキルは保護範囲、述語、項、付加部、骨格、曖昧さを整理する。

## 役割

- 変更してはいけない範囲を特定する。
- 文の中心になる述語を見つける。
- 述語に必要な項を確認する。
- 条件、理由、目的、時点、範囲などの付加部を切り分ける。
- 文の骨格を圧迫している付加部や冗長表現を見つける。
- 原文だけでは判断できない箇所を `ambiguities` として残す。

このスキルでは、原則として最終リライトを行わない。
ユーザーが分析だけを求めている場合は分析結果を出し、`natural-japanese-rewriter` の前段として使う場合は分析メモを内部入力として渡す。

## 分析手順

1. 対象範囲を確認する。
   - 日本語本文、見出し、箇条書き、コメント、UI 文言、エラーメッセージなど、リライト対象を特定する。
   - コードブロック、コマンド、JSON / YAML / TOML、表、URL、ファイルパス、API 名、設定値、数値、固有名詞、引用を保護する。

2. 述語を抽出する。
   - 「する」「返す」「検証する」「保存する」「有効である」など、文の中心になる動作や状態を拾う。
   - 複数の述語がある場合は、それぞれを分ける。

3. 項を確認する。
   - 各述語について、誰が、何を、何に、何へ、何から、何として、などの必須要素を確認する。
   - 省略された項が文脈から一意に復元できるか確認する。
   - 復元できない項を勝手に補わない。

4. 付加部を確認する。
   - 条件、理由、目的、時点、範囲、手段、例外、補足を切り分ける。
   - 付加部が長すぎて骨格を見えにくくしている場合は、リライト方針に文分割を入れる。

5. 骨格を整理する。
   - 文の中心を「誰が / 何が」「何を / 何に」「どうする / どうである」の形で把握する。
   - 骨格に関係しない説明が多い場合は、説明を後ろへ回すか文を分ける方針にする。

6. 曖昧さを残す。
   - 係り先が複数あり得る箇所、指示語の指す対象が不明な箇所、仕様判断が必要な箇所を記録する。
   - 曖昧さを本文で解消できない場合は、リライト側で `要確認` として扱う。

## 出力契約

次の YAML 形式で分析メモを作る。
前段として使う場合は、ユーザーへそのまま出さず、リライトの判断材料にする。

```yaml
protected_ranges:
  - 変更してはいけない語句、コード、数値、仕様、固有名詞、引用
rewrite_targets:
  - 不自然さ、翻訳調、冗長さ、係り受け、助詞、文分割の問題
predicates:
  - verb: 述語
    args:
      が: 主体または主題
      を: 対象
      に: 到達点、相手、状態など
    omitted_args:
      - 省略されているが復元できる項
adjuncts:
  - type: 条件、理由、目的、時点、範囲、手段、例外、補足
    text: 原文中の該当箇所
ambiguities:
  - 原文だけでは判断できない箇所
rewrite_policy:
  - 分割、語順変更、助詞調整、省略維持、明示化の方針
```

## 判断基準

- 項と付加部は、助詞だけで決めない。述語に必須か、文の意味を成立させるかで判断する。
- 省略された項は、読者が自然に一意復元できる場合だけ補完候補にする。
- 技術文では、自然さより仕様の正確さを優先する。
- 職務経歴書では、実績、役割、技術スタック、期間、数値を変えない。
- UI 文言やエラーメッセージでは、短さと明確さを優先する。
