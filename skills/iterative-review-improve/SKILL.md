# Iterative Review Improve

このスキルは、ユーザーが提示した `[入力]` に対して、レビューと改善を5回繰り返し、最終的な改善版を作るために使う。

## 入力

ユーザーから次のいずれかを受け取る。

- 直接貼り付けられた文章、プロンプト、仕様、計画、説明文
- ローカルファイルのパス
- 既存の出力物やドラフト

対象がファイルの場合は、該当ファイルを読み、ユーザーが編集を求めている場合のみファイルへ反映する。
対象がメッセージ本文の場合は、最終的な改善版を回答として提示する。

## 手順

1. 入力の目的、読者、制約、維持すべきトーンを把握する。
2. 入力に明示された条件を優先し、根拠のない内容追加や過度な意味変更を避ける。
3. 次の工程を正確に5回繰り返す。

```text
レビューする。
改善する。
```

各ループでは次を行う。

1. レビュー
   - 目的に対して弱い点を確認する。
   - 読みにくさ、曖昧さ、冗長さ、論理の飛躍、抜け漏れ、矛盾、トーンの不一致を確認する。
   - 技術文書や仕様の場合は、実装可能性、前提、境界条件、検証可能性も確認する。
2. 改善
   - レビューで見つけた問題を反映して入力を更新する。
   - 元の意図、事実、制約を保つ。
   - 改善のたびに文章を自然にし、不要な説明や重複を削る。

## 出力方針

原則として、最終回答では次を出す。

```markdown
## 改善後

{5回改善後の最終版}

## レビュー履歴

1回目: {主なレビュー観点と改善内容}
2回目: {主なレビュー観点と改善内容}
3回目: {主なレビュー観点と改善内容}
4回目: {主なレビュー観点と改善内容}
5回目: {主なレビュー観点と改善内容}
```

ただし、ユーザーが「最終版だけ」「ファイルに反映して」などの出力形式を指定した場合は、それを優先する。

## 判定ルール

- 5回より少なく終えない。
- 5回を超えて追加ループしない。ただし、ユーザーが明示的に追加改善を求めた場合は別タスクとして扱う。
- 各回で同じ指摘を繰り返すだけにしない。前回の改善結果を踏まえて、より細かい観点へ進める。
- 入力にない事実を断定しない。
- 意図が不明な箇所は、勝手に補完せず、必要に応じて「要確認」とする。
- コード、設定、仕様など正確性が重要な入力では、見た目の改善より意味の保持を優先する。
- 日本語入力は、原則として自然な日本語に整える。硬すぎる翻訳調やLLMらしい表現を避ける。

## 完了条件

- レビューと改善を5回実施している。
- 最終版が入力より読みやすく、目的に沿っている。
- 変更により元の意味や事実が壊れていない。
- ファイル編集を求められた場合は、指定ファイルが更新されている。
