# 要件定義書と実装計画書の node 紐づけ

要件定義書と実装計画書の各 node に UUID v7 を割り当て、
実装計画書の各 node に「どの要件を実装するか」を要件 node の UUID v7 で関連づけます。
これにより、要件 → 実装計画のトレーサビリティを機械的に追跡できるようにします。

## 入力

ユーザーから次の 2 つを受け取る。

- 要件定義書のファイルパス（Markdown を想定）
- 実装計画書のファイルパス（Markdown を想定）

パスが 1 つしか与えられない場合や、どちらの文書か判別できない場合は、
不足している方の提示を求める。
ファイルが読み取れない場合は、その旨を伝えて処理を中断する。

## node の定義

両文書に共通して、次を node として扱う。

- 見出し（`#`〜`######`）のうち、要件・機能・タスク・ステップを表すもの
- 要件項目・受け入れ基準・タスクを表す箇条書き項目（`- ` / `1. ` / `- [ ]` など）

文書のタイトル、目次、改訂履歴、前置きの説明文は node として扱わない。
どこまでを node とするか判断に迷う粒度（例: 箇条書きの孫階層まで含めるか）がある場合は、
見出しと第 1 階層の箇条書きまでを既定とし、処理後の報告でその旨を明記する。

文書に既存の要件 ID（`REQ-001`、`NFR-002` など）がある場合は、削除・置換せず併存させる。
既存 ID は対応関係の判定のヒントとして使う。

## UUID v7 の生成方法

高速な C 実装を優先し、次の順でフォールバックする。

1. **`uuidgen -7`**（util-linux / libuuid。C 実装で高速）
   - 事前に `uuidgen -7` を 1 回実行して動作確認する（util-linux 2.39 以降で利用可）
   - 複数個必要な場合は `for _ in $(seq N); do uuidgen -7; done` で一括生成する
2. **Python の `uuid.uuid7()`**（Python 3.14 以降）
   - `python3 -c "import uuid; print('\n'.join(str(uuid.uuid7()) for _ in range(N)))"`
3. **Python による自前実装**（上記が使えない場合）
   - RFC 9562 に従い、先頭 48 bit を Unix ミリ秒タイムスタンプ、
     version=7 / variant=10 を設定し、残りをランダムで埋める

必要な UUID の個数を先に数え、生成はまとめて 1 回のコマンドで行う
（node ごとにプロセスを起動しない）。

## UUID の埋め込み形式

Markdown の表示を壊さないよう、HTML コメントで埋め込む。

要件定義書の node:

```markdown
## ログイン機能 <!-- id: 019f8c89-a956-7576-8c4c-fd812ab141c1 -->

- パスワードは 8 文字以上とする <!-- id: 019f8c89-b012-7abc-9def-0123456789ab -->
```

実装計画書の node（`refs` に参照する要件 node の UUID をカンマ区切りで列挙）:

```markdown
### Step 1: ログイン API の実装 <!-- id: 019f8c8a-1111-7222-8333-444455556666, refs: 019f8c89-a956-7576-8c4c-fd812ab141c1 -->
```

- 既に `id:` が付与されている node には **再付与しない**（冪等性の確保）
- 既存の `refs:` は上書きせず、不足分の追加・誤りの指摘のみ行う

## 手順

1. **動作確認**: `uuidgen -7` の可否を確認し、使用する生成方法を決定する。
2. **要件定義書の解析**: node を抽出し、既存 `id` の有無を確認する。
3. **要件定義書への割り当て**: `id` 未付与の node の個数分 UUID v7 を一括生成し、埋め込む。
4. **実装計画書の解析**: node を抽出し、既存 `id` / `refs` の有無を確認する。
5. **対応関係の判定**: 実装計画書の各 node が、どの要件 node を実装・参照しているかを、
   次の優先順位で判定する。
   1. 計画 node が要件 ID（`REQ-001` など）や要件の見出し名を明示的に言及している → 確度「高」
   2. 機能名・用語・固有名詞が一致し、内容も矛盾しない → 確度「中」
   3. 章構成の対応や文脈からの推測 → 確度「要確認」
   - 1 つの計画 node が複数の要件 node に対応する場合は、複数の UUID を `refs` に列挙する
   - 対応する要件が見つからない計画 node は `refs` を付けず、「未対応」として報告する
   - 「要確認」の対応は `refs` に含めてよいが、報告で必ず根拠とともに明示する
6. **実装計画書への割り当て**: `id` と `refs` を埋め込む。
7. **検証**: `refs` に列挙した UUID がすべて要件定義書に実在することを grep で照合する。
   実在しない UUID が見つかった場合は修正し、修正の根拠が判断できない場合は
   その `refs` を外して「未対応」として報告する。
8. **報告**: 対応表と未対応 node の一覧を出力する。

## 出力フォーマット（報告）

```markdown
## トレーサビリティ対応表

| 実装計画 node | 参照する要件 node | 確度 |
| ------------- | ----------------- | ---- |
| Step 1: ログイン API の実装 | ログイン機能 / パスワードは 8 文字以上とする | 高 |
| Step 2: 画面実装 | ログイン機能 | 要確認 |

確度は「高」（明示的な言及あり）、「中」（用語の一致から妥当と判断）、
「要確認」（推測を含むためユーザーの確認が必要）の 3 段階で示す。

## 未対応の node

- 実装計画側: （対応する要件が見つからなかった計画 node。要件の漏れの可能性）
- 要件定義側: （どの計画 node からも参照されていない要件 node。実装漏れの可能性）

## 処理サマリ

- UUID 生成方法: uuidgen -7（libuuid）
- 要件定義書: 新規付与 N 件 / 既存 M 件
- 実装計画書: 新規付与 N 件 / refs 追加 M 件
```

## 注意事項

- 文書の本文・構造は変更しない。追加するのは HTML コメントのみ。
- UUID は再利用・使い回しをしない。node 1 つにつき新規生成した UUID を 1 つ割り当てる。
- 双方向の漏れ（参照されない要件、要件に紐づかない計画）は必ず報告する。
  これがこのスキルの主要な価値の 1 つである。
- 大きな文書では、埋め込み前に対象 node 数と方針を短く伝えてから編集する。

## 完了条件

- 両文書のすべての node に `id` が付与されている（既存 `id` の node は変更されていない）。
- 実装計画書の各 node に `refs` が付与されているか、「未対応」として報告されている。
- `refs` の全 UUID が要件定義書に実在することを grep で確認済みである。
- トレーサビリティ対応表、未対応 node の一覧、処理サマリが報告されている。
