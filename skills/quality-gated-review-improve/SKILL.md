# Quality Gated Review Improve

このスキルは、ユーザーが提示した `[入力]` に対して、レビュー、改善、採点を繰り返し、100点中80点以上、可能なら90点以上、さらに到達可能なら100点の品質まで引き上げるために使う。
質が下がる改善しか残っていない場合は、基準点に届いていなくてもループを終了し、その理由を報告する。
改善を重ねること自体を目的にせず、現在の最良版を守りながら、明確に品質が上がる変更だけを採用する。

## 入力

ユーザーから次のいずれかを受け取る。

- 直接貼り付けられた文章、プロンプト、仕様、計画、説明文
- ローカルファイルのパス
- 既存の出力物やドラフト
- 「この内容を80点以上にして」「品質を採点しながら改善して」のような改善対象の指定

ファイルパスだけを指定してこのスキルが呼ばれた場合は、そのファイルを改善して更新する依頼として扱う。
ただし、ユーザーが「レビューだけ」「提案だけ」「最終版を回答で出して」と指定した場合は、ファイルを書き換えず回答内に出す。

## 品質目標

- 最低目標: 100点中80点以上
- 合格目標: 100点中90点以上
- 最終目標: 到達可能なら100点
- 90点以上に到達した後の方針:
  - 100点を目指せそうなら、100点になるまでレビュー、改善、採点を続ける。
  - 100点にするのが厳しい場合は、90-99点の範囲で到達可能な最高点になるまでレビュー、改善、採点を続ける。
- 終了条件: 100点に到達した場合、または90-99点の範囲でこれ以上品質を上げられない場合
- 例外終了: 80点未満でも、残る改善案が品質を下げる、意味を壊す、根拠のない追加になる、またはユーザー条件に反する場合
- ループ上限: 原則として改善ループは最大8回まで。上限に達した場合は、その時点の最良版、最終点、残課題を報告して終了する。

## 採点基準

入力の種類に合わせて重みは調整してよいが、原則として次の観点で100点満点をつける。

- 目的適合性: 25点
  - 入力の目的、読者、用途に合っているか。
- 明確性: 20点
  - 曖昧さ、論理の飛躍、未定義の前提が少ないか。
- 構造: 15点
  - 見出し、順序、粒度、情報配置が使いやすいか。
- 実行可能性: 15点
  - 手順、条件、判定基準、完了条件が具体的か。
- 正確性・根拠保持: 15点
  - 元の意味、事実、制約、重要な用語が保たれているか。
- 仕上げ: 10点
  - 冗長さ、表記揺れ、トーン、読みやすさが整っているか。

採点は甘くしない。
点数だけでなく、減点理由と次に改善すべき点を短く書く。
採点は絶対評価として扱い、前回との差分だけで高くしない。
入力にない情報を足して見栄えを良くしただけの改善は、正確性・根拠保持で減点する。

## 手順

1. 入力を読む。
2. 入力の目的、読者、用途、制約、維持すべきトーンを把握する。
3. ファイルの場合は、既存の形式、メタデータ、見出し構造、コードブロック、リンクを壊さないようにする。
4. 初回レビューを行い、品質を100点満点で採点する。
5. 点数が100点なら終了する。
6. 点数が90点以上100点未満の場合は、100点を目指せるか確認する。
7. 100点を目指せそうなら、100点になるまでレビューと改善を続ける。
8. 100点にするのが厳しい場合は、90-99点の範囲で最高点になるまでレビューと改善を続ける。
9. 点数が80点以上90点未満の場合は、90点以上を目指せる明確な改善がある場合だけ改善する。
10. 点数が80点未満の場合は、レビューと改善を行い、再採点する。
11. 以後、品質目標に達するか、改善すると質が下がる状態になるまでループする。
12. 常に「現在の最良版」と「現在の最良点」を保持し、候補版が最良点を下回る場合は採用しない。

各ループでは次を行う。

1. レビュー
   - 現在の点数と減点理由を確認する。
   - 改善すれば点数が上がる箇所だけを候補にする。
   - 意味、事実、制約、トーンを壊す改善案は候補から外す。
2. 改善
   - レビューで見つけた有効な改善だけを反映する。
   - 元の意図、事実、制約、重要な用語を保つ。
   - 不要な説明、重複、抽象的すぎる表現を削る。
   - ファイル更新時は、最終的に採用された最良版だけを書き込む。
3. 採点
   - 改善後の品質を100点満点で採点する。
   - 前回より点数が下がった場合は、その改善を採用せず、直前の最良版に戻す。
   - 点数が変わらず、残る改善案も弱い場合はループを終了する。
   - 点数が上がっても、ユーザー条件から外れる変更が含まれる場合は採用しない。

## 改善をスキップする条件

次に当てはまる改善は行わない。

- 元の意味や事実を変えてしまう。
- 入力にない情報を断定的に追加する。
- ユーザーが指定したトーンや形式から外れる。
- 長くなるだけで目的適合性が上がらない。
- 表現は変わるが、点数が上がる見込みがない。
- コード、仕様、スキル定義などで、正確性や実行可能性を下げる。
- 採点基準の一部だけを上げる代わりに、別の重要な基準を大きく下げる。

## ループ終了条件

次のいずれかで終了する。

- 100点に到達した。
- 90-99点の範囲で、到達可能な最高点まで改善した。
- 80点以上90点未満に到達し、90点以上を目指す有効な改善候補がなくなった。
- 80点未満だが、残る改善案が品質を下げるものだけになった。
- 80点未満だが、入力不足によりこれ以上は推測で補うしかない。
- 前回から点数が上がらず、有効な改善候補がなくなった。
- 改善ループが8回に達した。

80点未満で終了する場合は、必ず理由を報告する。
8回上限で終了する場合も、上限到達であることと残課題を報告する。

## 出力方針

ユーザーがファイル更新を求めている場合、またはファイルパスだけでこのスキルが呼ばれた場合は、最終版をファイルへ反映する。
回答では、変更ファイル、最終点数、終了理由、主な改善内容を簡潔に報告する。
レビュー履歴は、全ループの詳細な全文ではなく、点数推移と主要な改善点に絞る。

ユーザーが回答内での提示を求めている場合は、原則として次を出す。

```markdown
## 改善後

{最終版}

## 採点

最終点: {点数}/100
終了理由: {100点に到達 / 90-99点の範囲で最高点に到達 / 80点以上だが改善余地なし / 入力不足など}

## レビュー履歴

1回目: {点数}/100 - {主な減点理由と改善内容}
2回目: {点数}/100 - {主な減点理由と改善内容}
...
```

入力が長い場合は、最終版の全文を出すより、ファイル更新または要点報告を優先する。
ただし、ユーザーが「全文を出して」と指定した場合は全文を提示する。

## 判定ルール

- 点数が80点未満の場合は、原則としてレビューと改善を続ける。
- ただし、質が下がる改善しかできない場合は停止する。
- 80点以上でも、90点以上を狙える明確な改善があれば続ける。
- 90点以上に到達しても即終了しない。100点を目指せそうなら100点まで続ける。
- 100点にするのが厳しい場合は、90-99点の範囲で最高点になるまで続ける。
- 100点に到達したら、不要な改善で品質を下げないよう終了する。
- 点数を上げるためだけに、入力の目的から外れた機能や説明を追加しない。
- 入力にない事実を断定しない。
- 意図が不明な箇所は、勝手に補完せず、必要に応じて「要確認」とする。
- コード、設定、仕様、スキルなど正確性が重要な入力では、見た目の改善より意味の保持を優先する。
- 日本語入力は、原則として自然な日本語に整える。硬すぎる翻訳調やLLMらしい表現を避ける。
- ファイル編集時は、ユーザーの既存変更を巻き戻さない。

## 検証

対象に応じて、可能な範囲で確認する。

- Markdown / AsciiDoc: 見出し、リスト、コードブロック、表の崩れがないか確認する。
- スキル: `quick_validate.py` など利用可能なバリデータがあれば実行する。
- コードや設定: 既存のプロジェクトコマンドがあれば、最小限の検証を行う。
- 検証できない場合は、理由を最後に報告する。

## 完了条件

- レビュー、改善、採点のループが実施されている。
- 最終点数と終了理由が明確になっている。
- 採用された最終版が、ループ中の最良版である。
- 90点以上に到達した場合は、100点を目指したか、100点が厳しいため90-99点の範囲で最高点を採用したことが分かる。
- 最終版が入力より読みやすく、目的に沿っている。
- 変更により元の意味や事実が壊れていない。
- 80点未満で終了する場合は、質を下げる改善しか残っていない理由、または入力不足の理由が報告されている。
- ファイル編集を求められた場合、またはファイルパスだけで呼ばれた場合は、指定ファイルが更新されている。
- 可能な範囲で検証し、結果を報告している。
