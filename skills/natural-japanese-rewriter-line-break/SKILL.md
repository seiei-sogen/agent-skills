# Natural Japanese Rewriter Line Break

## 目的

LLMくさい日本語、翻訳調の日本語、不自然に硬い日本語、文意が取りづらい日本語を、原意を保ったまま自然で読みやすい日本語に修正する。

対象は、README、仕様書、技術記事、コメント、UI 文言、エラーメッセージ、issue、PR 説明文などの日本語テキスト。

この skill の目的は、単なる言い換えではない。

文の骨格を確認し、読者が「誰が、何を、何に、どうするのか」を自然に追えるようにする。

---

# 最優先ルール

自然さよりも、次を優先する。

1. 原意を変えない。
2. 事実を変えない。
3. 仕様を変えない。
4. 技術的な意味を変えない。
5. コード、識別子、コマンド、URL、ファイル名を勝手に変えない。
6. 不明な内容を推測で補わない。
7. 問題がない文は無理に直さない。

文章を自然にするためであっても、仕様・事実・意図を勝手に補完してはいけない。

---

# 意味保持で特に守るもの

修正時は、次を壊していないか必ず確認する。

- 否定
- 条件
- 例外
- 因果関係
- 時系列
- 範囲
- 主体
- 対象
- 能動 / 受け身
- 可能性
- 義務
- 許可
- 推量
- 断定の強さ
- 技術用語の意味
- 仕様上の制約

## 可能性を断定に変えない

修正前:

> この API は、条件に一致しない場合にエラーを返すことがあります。

危険な修正:

> この API は、条件に一致しない場合にエラーを返します。

理由:

- 「返すことがあります」が「返します」になると、可能性が断定に変わる。

安全な修正:

> この API は、条件に一致しない場合、エラーを返すことがあります。

または、原文が十分に自然なら直さない。

仕様として常に返すことが分かっている場合だけ、次のように断定してよい。

> この API は、条件に一致しない場合にエラーを返します。

# 中心方針: 述語を中心に、項と付加部を分ける

悪文を直すときは、文末だけを見るのではなく、文中の述語ごとに構造を確認する。

## 述語

述語は、文や節の中心になる要素である。

例:

- 読み込む
- 検証する
- 返す
- 表示する
- 失敗する
- 有効である
- 必要になる
- 対応している

述語は文末にあるとは限らない。

連体修飾節、条件節、理由節、並列節の中にも述語がある。

例:

> ユーザーが入力した値を検証したあと、結果を保存します。

この文には、少なくとも次の述語がある。

- 入力した
- 検証した
- 保存します

---

## 項

項は、述語が要求する要素である。

その述語の意味を成立させるために必要な参加者、対象、到達点、起点、結果などを指す。

例:

> ユーザーがファイルをサーバーにアップロードする。

述語「アップロードする」は、次のような項を取りやすい。

- 誰が: ユーザーが
- 何を: ファイルを
- どこに: サーバーに

ただし、日本語では項を省略できる。

省略してよいかどうかは、文脈から一意に復元できるかで判断する。

---

## 付加部

付加部は、述語が必ず要求するわけではない補足情報である。

例:

- 昨日
- 会議で
- 急な仕様変更により
- 必要に応じて
- 手動で
- 管理画面から
- エラー発生時に

付加部は、時・場所・理由・条件・手段・様態などを表すことが多い。

ただし、付加部だから不要という意味ではない。

仕様上重要な条件や理由は残す。

問題は、付加部が長すぎたり、項と混ざったりして、文の骨格を読みにくくする場合である。

---

## 注意: 助詞や意味分類だけで項 / 付加部を決めない

「に」「で」「から」「として」などは、項にも付加部にも使われる。

次の要素は、述語によっては項になる。

```text
東京に住む
DBから取得する
サーバーに接続する
AをBに変換する
ユーザーを管理者として扱う
```

この場合、次の要素は単なる飾りではなく、述語の意味を成立させる重要な要素である。

- 東京に
- DBから
- サーバーに
- Bに
- 管理者として

項か付加部かは、助詞だけでなく、述語ごとの格フレームで判断する。

---

## 骨格

骨格は、文の中心になる「項 + 述語」の組み合わせである。

例:

> この関数は設定ファイルを読み込みます。

骨格は、次の組み合わせである。

```text
この関数が / 設定ファイルを / 読み込む
```

悪文を直すときは、まずこの骨格を見つける。

骨格が見えない文は、読み手が「誰が何をどうするのか」を復元しなければならず、読みにくくなる。

---

# 改行と段落

読みやすさを保つため、原則として一文ごとに改行する。

段落の区切りは空行で示す。

## 基本ルール

- 一文ごとに改行する。
- 段落が変わる箇所には空行を入れる。
- 複数の内容を一文に詰め込まず、必要なら文を分ける。
- 文のまとまりが変わる場合は、改行だけでなく空行で段落を分ける。
- 一文ごとの改行によって、主語・述語、項・述語、条件・結果の対応を追いやすくする。
- Markdown、Asciidoc など、HTML にレンダリングできるフォーマットでは、ブラウザ表示でも一文ごとに必ず改行されるようにする。
- HTML では、ブラウザ表示でも一文ごとに必ず改行されるようにする。
- Markdown、Asciidoc、HTML では、一文の終わりごとに `<br>` タグを入れて改行する。
- HTML で、対象の文がすでに `<p>` タグなどのブロック要素によって段落として改行されている場合は、追加で `<br>` タグを入れない。
- `<br>` タグは、文末の句点・疑問符・感嘆符の直後に置く。
- すでに文末に `<br>` タグがある場合は、重複して追加しない。
- 本文に文字列として `\n` を書かない。
- ただし、文字列リテラル、JSON、テストデータなどで `\n` が必要な場合は変更しない。

## 例

修正前:

> この関数は設定ファイルを読み込み、設定値を検証します。検証に失敗した場合は、エラーを返します。設定ファイルが存在しない場合は、デフォルト値を使用します。

修正後:

> この関数は設定ファイルを読み込み、設定値を検証します。<br>
> 検証に失敗した場合は、エラーを返します。<br>
> 設定ファイルが存在しない場合は、デフォルト値を使用します。<br>

段落を分ける場合:

> この関数は設定ファイルを読み込み、設定値を検証します。<br>
> 検証に失敗した場合は、エラーを返します。<br>
>
> 設定ファイルが存在しない場合は、デフォルト値を使用します。<br>

HTML で `<p>` タグにより段落として改行されている場合:

```html
<p>この関数は設定ファイルを読み込み、設定値を検証します。</p>
<p>検証に失敗した場合は、エラーを返します。</p>
```

この場合は、各文末に追加で `<br>` を入れない。

## 例外

次の場合は、無理に一文ごとの改行を適用しない。

- コードブロック
- 表
- inline code
- コマンド例
- JSON / YAML / TOML などの構造化データ
- 自動生成された内容
- Markdown の構造を壊す箇所
- UI 文言など、短い断片として扱う方が自然な箇所
- 既存プロジェクトのフォーマット規約と衝突する箇所

Markdown や Asciidoc の箇条書きでは、1項目に複数文が含まれる場合も、可能なら一文ごとに `<br>` タグを入れて改行する。

ただし、階層やリスト構造を壊さない。

修正前:

```md
- 設定ファイルを読み込みます。存在しない場合はデフォルト値を使用します。
```

修正後:

```md
- 設定ファイルを読み込みます。<br>
  存在しない場合は、デフォルト値を使用します。<br>
```

内容を分けた方が読みやすい場合:

```md
- 設定ファイルを読み込みます。<br>
- 存在しない場合は、デフォルト値を使用します。<br>
```

# 基本方針

## 1. 項は原則として述語の近くに置く

述語が要求する項は、できるだけ述語の近くに置く。

ただし、次の場合は文頭や離れた位置にあってもよい。

- 主題として自然に提示されている。
- 対比されている。
- すでに話題になっている情報を受けている。
- UI 文言や見出しとして短く提示されている。
- 文脈上、係り先が明確である。

つまり、次を基本にする。

```text
項は述語の近くへ。
ただし、自然な主題化や対比は無理に崩さない。
```

例:

> この設定は、管理画面から変更できます。

「この設定」は述語「変更できます」の対象だが、主題として文頭にあるため自然である。

このような文は無理に直さない。

---

## 2. 項の欠落は、復元できない場合だけ補う

日本語では、主語や目的語を省略するのが自然な場合がある。

項が表面に出ていないだけで、機械的に補ってはいけない。

確認すること:

- 述語がどの項を要求しているか。
- その項は文中にあるか。
- 省略されている場合、文脈から一意に復元できるか。
- 復元できない場合、本文で補える根拠があるか。

修正前:

> 導入により改善した。

問題:

- 「改善した」の対象が不明である。
- 何が改善したのか、文脈から復元できない場合は曖昧である。

文脈から対象が分かる場合:

> 導入により、起動時間が改善した。

文脈から対象を判断できない場合:

> 導入により改善した。

作業報告:

> 要確認: 「何が改善したのか」を原文だけでは判断できませんでした。

---

## 3. 省略してよいかは、述語の項が文脈で満たされているかで判断する

次のような文は、文脈によっては自然である。

> 検討します。

直前に対象が明示されていれば問題ない。

例:

> この案は採用できますか？<br>
> 検討します。

この場合、「何を検討するか」は復元できる。

無理に次のように補うと、不自然になることがある。

> 私たちはこの案を検討します。

一方で、文脈がない場所では曖昧である。

> 検討します。

問題:

- 誰が検討するのか。
- 何を検討するのか。
- いつまでに検討するのか。

文脈から分かる場合だけ補う。

> 開発チームで、この仕様変更を検討します。

「開発チーム」や「仕様変更」が文脈から分からない場合は補わない。

---

## 4. 付加部が骨格を圧迫している場合は分ける

長い理由、条件、時系列、補足説明が、項と述語の間に割り込むと読みにくくなる。

修正前:

> 昨日、急な仕様変更があったため十分なテストができないまま、私たちはリリースした。

骨格:

> 私たちはリリースした。

付加部・補足:

- 昨日
- 急な仕様変更があったため
- 十分なテストができないまま

修正後:

> 私たちは昨日リリースした。<br>
> 急な仕様変更があったため、十分なテストはできていなかった。

理由:

- 骨格「私たちはリリースした」を先に出した。
- 「急な仕様変更があったため」が「十分なテストはできていなかった」にかかるように整理した。
- 因果関係を勝手に変えていない。

---

## 5. 項と付加部が同じ助詞で紛れる場合は、語順や表現を変える

日本語では、「に」「で」「から」「として」などが、項にも付加部にも使われる。

助詞だけでは関係がはっきりしない場合がある。

修正前:

> 会議で部長に資料で説明した。

考えられる構造:

- 会議で: 場所の付加部
- 部長に: 「説明した」の相手を表す項
- 資料で: 手段の付加部
- 説明した: 述語

修正後:

> 会議で、資料を使って部長に説明した。

必要なら:

> 会議で、資料を使って部長に内容を説明した。

ただし、「内容」が文脈から分からない場合は補わない。

---

## 6. 項の係り先が遠い場合は、節境界を明示する

日本語は語順の自由度が高いため、項と述語が離れると読みにくくなる。

修正前:

> 太郎が花子が次郎に渡した本を読んだ。

問題:

- 「太郎が」と「花子が」が続き、一瞬構造を取りづらい。
- 「太郎が」がどの述語に対応するのか迷いやすい。

修正後:

> 太郎は、花子が次郎に渡した本を読んだ。

または:

> 花子が次郎に渡した本を、太郎が読んだ。

方針:

- 主節の項と従属節の項を分ける。
- 必要なら読点を入れる。
- 長い連体修飾は文に分ける。
- 主題化できる場合は「は」を使って主節の骨格を見せる。

---

# 修正観点

## 1. 主語と述語の対応を分かりやすくする

主語と述語の対応を見るときも、単に「主語があるか」だけで判断しない。

まず、述語ごとに必要な項を確認する。

確認すること:

- どの述語の主語なのか。
- 主節の主語か、従属節の主語か。
- 主語が省略されていても文脈から分かるか。
- 主語を補うことで不自然にならないか。
- 主語と述語が遠すぎないか。

修正方針:

- 主語と述語を近づける。
- 主語が切り替わる箇所では明示する。
- 文脈で明らかな主語は省略してよい。
- 長い付加部が主語と述語の間にある場合は、文を分ける。

例:

修正前:

> この API は、ユーザーが送信したリクエストに含まれるパラメータを検証したうえで、必要なデータをデータベースから取得し、レスポンスとして返すために使用されます。

修正後:

> この API は、リクエストのパラメータを検証し、必要なデータをデータベースから取得して返します。

理由:

- 「この API は」と「返します」の対応を近づけた。
- 「使用されます」を避け、API の動作を直接書いた。
- 長い修飾を整理した。

---

## 2. 目的語と動詞の対応を分かりやすくする

目的語は、述語が要求する重要な項であることが多い。

「何をどうするのか」が曖昧な場合は、目的語と動詞を近づける。

確認すること:

- 「を」項がどの述語に対応するか。
- 目的語が省略されていても復元できるか。
- 複数の目的語が並んで読みにくくないか。
- 「処理する」「対応する」「行う」などで動詞が曖昧になっていないか。

修正方針:

- 「何を」「どうする」を近づける。
- 抽象的な動詞を具体的な動詞に変える。
- 目的語が複数ある場合は文を分ける。
- 目的語を補う場合は、文脈から確実に分かる範囲に限る。

例:

修正前:

> 設定ファイルの読み込みとバリデーションを行います。

修正後:

> 設定ファイルを読み込み、内容を検証します。

技術用語として「バリデーション」を残す必要がある場合:

> 設定ファイルを読み込み、内容をバリデーションします。

注意:

- 技術用語を一般語に変えて意味がずれる場合は、無理に言い換えない。

---

## 3. 助詞を自然にする

助詞は、項と付加部の関係を示す重要な手がかりである。

ただし、日本語の助詞は項と付加部を完全には区別しない。

確認すること:

- 「は」と「が」の使い分けが自然か。
- 「を」「に」「で」が、どの述語に対応しているか。
- 同じ助詞が続いて読みにくくなっていないか。
- 「について」「に関して」「において」が多すぎないか。
- 助詞だけでは項と付加部の区別が曖昧でないか。

修正方針:

- 項は述語の近くへ置く。
- 付加部は外側へ出すか、別文にする。
- 手段の「で」は「〜を使って」に変えると明確になる場合がある。
- 場所の「で」は「〜では」「〜から」に変えると自然になる場合がある。
- 「〜において」は、自然なら「〜で」「〜では」にする。
- 「〜に関して」「〜について」は、不要なら削る。

例:

修正前:

> この処理において、ユーザーに対して通知を送信します。

修正後:

> この処理では、ユーザーに通知を送信します。

さらに自然な場合:

> この処理では、ユーザーに通知します。

ただし、「通知を送信する」という仕様表現が必要なら残す。

---

## 4. 曖昧な部分を明確にする

曖昧さは、項の欠落、係り先の遠さ、付加部の割り込みから生じることが多い。

確認すること:

- 「これ」「それ」「この」「その」「上記」「以下」の対象が明確か。
- 省略された項を一意に復元できるか。
- 条件の範囲が明確か。
- 理由がどの述語にかかるか明確か。
- 修飾語がどの名詞・述語にかかるか明確か。
- 「適切に」「必要に応じて」「柔軟に」が具体的な意味を持っているか。

曖昧さの扱い:

1. 文脈から一意に分かる曖昧さは、明確に書き換える。
2. 複数候補がある曖昧さは、本文では断定せず、作業報告に「要確認」と書く。
3. 曖昧だが実害がない表現は、無理に直さない。

例:

修正前:

> これを有効にすると、適切に処理されます。

文脈から「これ」が自動保存設定だと分かる場合:

> 自動保存を有効にすると、変更内容が自動で保存されます。

文脈から判断できない場合:

> この設定を有効にすると、処理内容が変わります。

作業報告:

> 要確認: 「これ」と「適切に処理されます」の具体的な内容は、原文だけでは判断できませんでした。

---

## 5. 補完して不自然になる場合は、自然な省略を使う

日本語では、文脈上明らかな項を省略した方が自然な場合がある。

英語のようにすべての主語・目的語を明示すると、かえって不自然になる。

判断基準:

- 述語の項が文脈で満たされているか。
- 読者が省略された項を一意に復元できるか。
- 省略しても誤読されないか。
- 主語や目的語を補ったせいで、くどくなっていないか。
- 主語が切り替わる箇所では、必要に応じて明示されているか。

修正方針:

- 文脈で明らかな項は省略してよい。
- 復元できない項は補うか、要確認にする。
- 同じ主語を連続して繰り返さない。
- 技術文書では、誤読が生じる省略は避ける。

例:

修正前:

> この関数は設定ファイルを読み込みます。この関数は設定値を検証します。この関数は検証に失敗した場合にエラーを返します。

修正後:

> この関数は設定ファイルを読み込み、設定値を検証します。<br>
> 検証に失敗した場合は、エラーを返します。

理由:

- 2文目以降の「この関数は」は省略しても復元できる。
- 省略した方が自然。
- 「検証に失敗した場合」と「エラーを返します」の関係が明確。

---

## 6. 長大な文意を分かりづらくする修飾を避ける

長い連体修飾や理由節は、多くの場合、付加部や補足情報が骨格を圧迫している状態である。

確認すること:

- 名詞の前に長い修飾が続いていないか。
- 付加部が項と述語の間に割り込んでいないか。
- 条件、理由、結果が一文に詰め込まれていないか。
- 読点だけで無理につないでいないか。
- 主節の骨格がすぐ分かるか。

修正方針:

- 長い修飾は文に分ける。
- 条件、理由、結果を整理する。
- 骨格を先に出す。
- 重要な結論を文末まで引き延ばさない。
- 項は述語の近くへ、付加部は外側へ置く。
- ただし、自然な主題化や対比は維持する。

例:

修正前:

> ユーザーが入力した値が設定ファイルで定義されている条件に一致しない場合に発生するエラーを表示する処理を追加します。

修正後:

> ユーザーが入力した値を、設定ファイルで定義された条件と照合します。<br>
> 条件に一致しない場合は、エラーを表示します。

理由:

- 「エラーを表示する処理」という長い名詞句を分解した。
- 「入力値を照合する」「エラーを表示する」という述語中心の構造にした。
- 条件と結果を分けた。

---

# LLMくさくなりやすい表現

次の表現は、必要がない場合だけ修正する。

常に禁止ではない。

- 〜することができます
- 〜することが可能です
- 〜する形になります
- 〜していきます
- 〜を行います
- 〜を実施します
- 〜において
- 〜に関して
- 〜については
- 適切に
- 柔軟に
- 効率的に
- シームレスに
- ユーザー体験を向上させます
- 〜する必要があります
- 〜を提供します
- 活用します
- さまざまな
- 多様な
- 包括的な

## 「〜することができます」

文脈によって意味が違うため、安易に断定形へ変えない。

能力・可能性を表す場合:

> 〜できます

仕様として対応している場合:

> 〜に対応しています

実際の動作を表す場合:

> 〜します

注意:

- 「できます」を安易に「します」へ変えない。
- 可能性、権限、仕様上の対応を、断定的な動作に変えない。

## 「〜を行います」

修正前:

> 入力値の確認を行います。

修正後:

> 入力値を確認します。

ただし、専門的な処理名として「確認処理」が必要な場合は、無理に崩さない。

## 「適切に」

修正前:

> エラーを適切に処理します。

具体的な処理が分かる場合:

> エラーをログに記録します。

または:

> エラーをユーザーに表示します。

具体的な処理が分からない場合:

> エラーを処理します。

作業報告:

> 要確認: 「適切に」が指す具体的な処理内容は判断できませんでした。

# 作業手順

## Step 1: 対象範囲を確認する

修正対象を確認する。

- 日本語本文
- コメント
- ドキュメント
- UI 文言
- エラーメッセージ
- Markdown の本文
- コードブロック外の説明文

コード、設定値、識別子、コマンドは原則として対象外である。

ただし、ユーザーが明示的に求めた場合は、コードコメント、docstring、エラーメッセージなどの日本語だけを修正してよい。

---

## Step 2: 保護対象を確認する

変更してはいけないものを先に確認する。

- コード
- 変数名
- 関数名
- クラス名
- API 名
- コマンド
- オプション名
- 設定値
- ファイル名
- ディレクトリ名
- URL
- Markdown のリンク先
- inline code
- コードブロック
- JSON / YAML / TOML などのキー
- テストスナップショット
- 自動生成ファイル

これらは、明確な指示がない限り変更しない。

---

## Step 3: 文体を確認する

周囲の文体に合わせる。

- です・ます調
- だ・である調
- 常体
- UI 向けの短い文
- README 向けの説明文
- 仕様書向けの正確な文
- コメント向けの簡潔な文

文体が混在している場合は、主な文体に合わせる。

ただし、文体統一のために意味を変えない。

---

## Step 4: 述語を特定する

文ごと、節ごとに述語を拾う。

確認すること:

- 主節の述語
- 従属節の述語
- 連体修飾節の述語
- 条件節の述語
- 理由節の述語
- 並列されている述語

文末だけを見て判断しない。

例:

> ユーザーが入力した値を検証し、問題がなければ保存します。

述語:

- 入力した
- 検証し
- なければ
- 保存します

中心になる述語:

- 検証する
- 保存する

---

## Step 5: 述語ごとに項を書き出す

述語ごとに、必要な項を確認する。

形式:

```text
述語: <述語>
項:
- 誰が / 何が
- 何を
- 誰に / 何に
- どこへ / どこに
- 何から
- 何として
- 何に変化する / 何へ変換する
付加部:
- いつ
- どこで
- なぜ
- どの条件で
- どの手段で
- どのように
```

例:

> 管理者がユーザーにメールで通知する。

```text
述語: 通知する
項:
- 誰が: 管理者が
- 誰に: ユーザーに
付加部:
- 手段: メールで
```

ただし、「メールを通知する」のように言い換えると不自然な場合があるため、目的語を機械的に補わない。

---

## Step 6: 項があるか、復元できるか確認する

各述語について、項が次のどれかを確認する。

1. 文中に明示されている。
2. 文脈から一意に復元できる。
3. 複数候補がある。
4. 復元できない。

対応:

- 1 の場合は、係り先が分かりやすい位置にあるか確認する。
- 2 の場合は、省略して自然なら省略したままにする。
- 3 の場合は、本文では断定せず、要確認にする。
- 4 の場合は、根拠があれば補う。
- 4 の場合でも、根拠がなければ要確認にする。

---

## Step 7: 付加部が骨格を邪魔していないか確認する

付加部について確認する。

- 長すぎないか。
- 項と述語の間に割り込んでいないか。
- どの述語にかかるか明確か。
- 条件や理由の範囲が明確か。
- 複数の付加部が同じ助詞で並んでいないか。

問題がある場合:

- 付加部を文頭に出す。
- 付加部を文末に回す。
- 付加部を別文にする。
- 「〜で」を「〜を使って」「〜では」「〜から」などに変える。
- 条件、理由、結果を分ける。

ただし、付加部が仕様上重要な条件や制約を表す場合は削らない。

---

## Step 8: 項は内側へ、付加部は外側へ置く

語順で、項と付加部の違いを読み手に見せる。

基本方針:

```text
[付加部] [主題/主語] [項] [述語]
```

または、

```text
[主題/主語] [付加部] [項] [述語]
```

重要なのは、述語が要求する項と述語の関係を読み手が迷わないことである。

ただし、このルールは機械的に適用しない。

次のような文は、自然なら残す。

> この設定は、管理画面から変更できます。

「この設定」は述語「変更できます」の対象だが、主題として文頭にあるため自然である。

---

## Step 9: 自然な日本語に整える

構造を直したあとで、自然さを整える。

確認すること:

- 同じ語を繰り返しすぎていないか。
- 主語を補いすぎていないか。
- 「〜することができます」が続いていないか。
- 「〜を行う」が多くないか。
- 漢語が続きすぎていないか。
- 読点だけで長くつないでいないか。
- 文体が揃っているか。
- 一文ごとに改行しているか。
- Markdown、Asciidoc、HTML では、一文の終わりごとに `<br>` タグを入れているか。
- 段落の区切りを空行で示しているか。

注意:

- 自然さのために意味を変えない。
- 少し硬くても正確な文は、無理に崩さない。
- 問題がない文は変更しない。

---

## Step 10: 変更後に検査する

修正後、必ず確認する。

- 原文の意味を変えていないか。
- 事実を足していないか。
- 仕様を変えていないか。
- 否定、条件、例外、因果関係を壊していないか。
- 可能性、義務、許可、推量を変えていないか。
- 述語ごとの項が自然に対応しているか。
- 省略された項は文脈から復元できるか。
- 項と付加部が混ざっていないか。
- 主語と述語が対応しているか。
- 目的語と動詞が対応しているか。
- 助詞が自然か。
- 指示語の対象が明確か。
- 不要な補完をしていないか。
- 自然な省略ができているか。
- 長すぎる修飾が残っていないか。
- 一文ごとに改行しているか。
- Markdown、Asciidoc、HTML では、一文の終わりごとに `<br>` タグを入れているか。
- 段落の区切りを空行で示しているか。
- `<br>` タグの追加によって Markdown、Asciidoc、HTML の構造を壊していないか。
- コードや識別子を壊していないか。
- Markdown の構造を壊していないか。

---

# ファイル編集時のルール

ファイルを直接修正する場合は、次を守る。

- 変更は最小限にする。
- 日本語として問題ない文は変更しない。
- 一文ごとに改行する。
- Markdown、Asciidoc、HTML では、一文の終わりごとに `<br>` タグを入れて改行する。
- HTML で、対象の文がすでに `<p>` タグなどのブロック要素で改行されている場合は、追加で `<br>` タグを入れない。
- 段落の区切りは空行で示す。
- 一文ごとの改行や `<br>` タグの追加によって Markdown、Asciidoc、HTML の構造が壊れる場合は、構造を優先する。
- Markdown の見出し構造を壊さない。
- 箇条書きの階層を壊さない。
- 表の列数を変えない。
- コードブロック内は原則として変更しない。
- inline code の中身は変更しない。
- リンク先 URL は変更しない。
- 仕様が変わる修正はしない。
- 自動生成ファイルは原則として変更しない。
- 判断できない箇所は、本文に勝手に書かず、作業報告に出す。

---

# 出力形式

## 文章だけを修正する場合

```text
## 修正案

<自然な日本語に直した文章>

## 主な修正点

- 述語ごとに項と付加部を整理
- 主語と述語の対応を整理
- 目的語と動詞の対応を明確化
- 助詞を修正
- 曖昧な指示語を具体化
- 不自然な補完を避け、自然に省略
- 長い付加部や修飾を分割
- 一文ごとに改行し、段落を空行で分割

## 要確認

<原文だけでは判断できない点があれば書く>
```

要確認がない場合は、無理に項目を作らなくてよい。

## ファイルを編集した場合

```text
## 変更内容

- <変更した内容>
- <変更した内容>

## 変更しなかった内容

- <意図的に残した表現があれば書く>

## 要確認

- <判断できなかった点があれば書く>
```

変更しなかった内容や要確認がない場合は、無理に項目を作らなくてよい。

# 良い修正の基準

良い修正は、次を満たす。

- 原文の意味が保たれている。
- 技術的な意味が変わっていない。
- 述語ごとの項が分かりやすい。
- 項と述語の対応が分かりやすい。
- 付加部が骨格を邪魔していない。
- 主語と述語の対応が分かりやすい。
- 目的語と動詞の対応が分かりやすい。
- 助詞が自然である。
- 曖昧な箇所が減っている。
- 必要以上に説明しすぎていない。
- 日本語として自然な省略がある。
- 長い修飾が整理されている。
- 一文ごとに改行されている。
- 段落の区切りが空行で示されている。
- 文が読みやすい。
- 変更量が必要最小限である。

---

# 悪い修正の基準

悪い修正は、次のようなもの。

- 原文より意味が広がっている。
- 原文より意味が狭くなっている。
- 仕様を勝手に補っている。
- 能力表現を断定表現に変えている。
- 可能性を義務に変えている。
- 受け身を能動に変えて主体を変えている。
- 省略可能な項まで機械的に補っている。
- すべての主語を明示して不自然になっている。
- 付加部を削りすぎて必要な条件が消えている。
- 「自然さ」のために技術的な正確さを落としている。
- 曖昧な内容を推測で具体化している。
- 技術用語まで不要に言い換えている。
- コード、識別子、コマンドを壊している。
- Markdown の構造を壊している。
- 一文ごとの改行を優先して、表やコードブロックを壊している。
- 問題のない文まで書き換えている。

# 典型例

## 例1: 項の欠落

修正前:

> 導入により改善した。

問題:

- 「改善した」の対象項がない。
- 何が改善したのか分からない。

文脈から対象が分かる場合:

> 導入により、起動時間が改善した。

対象が分からない場合:

> 導入により改善した。

作業報告:

> 要確認: 「何が改善したのか」を特定できませんでした。

---

## 例2: 項と付加部が同じ助詞で並ぶ

修正前:

> 会議で部長に資料で説明した。

修正後:

> 会議で、資料を使って部長に説明した。

説明:

- 「部長に」は「説明した」の項である。
- 「会議で」「資料で」は付加部である。
- 手段の「資料で」を「資料を使って」にして、係り先を明確にした。

---

## 例3: 項の係り先が遠い

修正前:

> 太郎が花子が次郎に渡した本を読んだ。

修正後:

> 太郎は、花子が次郎に渡した本を読んだ。

または:

> 花子が次郎に渡した本を、太郎が読んだ。

説明:

- 主節の項と従属節の項を分けた。
- 二重の「が」による読みにくさを減らした。

---

## 例4: 長い付加部が骨格を圧迫している

修正前:

> 昨日、急な仕様変更があったため十分なテストができないまま、私たちはリリースした。

修正後:

> 私たちは昨日リリースした。<br>
> 急な仕様変更があったため、十分なテストはできていなかった。

説明:

- 骨格「私たちはリリースした」を先に出した。
- 理由を別文にした。
- 「急な仕様変更」が「十分なテストはできていなかった」にかかるように整理した。

---

## 例5: 場所が項になる場合

修正前:

> サーバーに接続し、データを取得します。

この文の「サーバーに」は、単なる場所ではなく「接続する」の項である。

削ると意味が変わる。

悪い修正:

> 接続し、データを取得します。

理由:

- どこに接続するのかが消えている。

安全な判断:

> サーバーに接続し、データを取得します。

問題がなければ修正しない。

---

## 例6: 冗長な可能表現

修正前:

> この機能を使用することで、ユーザーはファイルを簡単にアップロードすることができます。

修正後:

> この機能を使うと、ユーザーはファイルを簡単にアップロードできます。

さらに短くできる場合:

> この機能で、ユーザーはファイルを簡単にアップロードできます。

---

## 例7: 意味が変わるため断定しない

修正前:

> この API は、条件に一致しない場合にエラーを返すことがあります。

危険な修正:

> この API は、条件に一致しない場合にエラーを返します。

安全な修正:

> この API は、条件に一致しない場合、エラーを返すことがあります。

仕様として常に返すことが分かっている場合だけ:

> この API は、条件に一致しない場合にエラーを返します。

---

## 例8: 「〜を行う」を直す

修正前:

> 入力値の確認を行い、問題がない場合は保存処理を行います。

修正後:

> 入力値を確認し、問題がなければ保存します。

---

## 例9: 助詞を直す

修正前:

> この設定により、アプリケーションにおいてキャッシュを有効にします。

修正後:

> この設定で、アプリケーションのキャッシュを有効にします。

---

## 例10: 指示語を明確にする

修正前:

> これを無効にすると、処理が変わります。

「これ」が自動更新だと分かる場合:

> 自動更新を無効にすると、更新処理が変わります。

対象が分からない場合:

> この設定を無効にすると、処理が変わります。

作業報告:

> 要確認: 「これ」が指す設定を特定できませんでした。

---

## 例11: 自然な省略を使う

修正前:

> このコマンドはビルドを実行します。このコマンドは成果物を dist に出力します。このコマンドは失敗時にエラーを表示します。

修正後:

> このコマンドはビルドを実行し、成果物を dist に出力します。<br>
> 失敗した場合は、エラーを表示します。

説明:

- 2文目以降の「このコマンドは」は省略しても復元できる。
- 項が文脈で満たされているため、省略した方が自然。

---

## 例12: 一文ごとに `<br>` タグを入れて改行する

修正前:

```md
この関数は設定ファイルを読み込みます。存在しない場合は、デフォルト値を使用します。
```

修正後:

```md
この関数は設定ファイルを読み込みます。<br>
存在しない場合は、デフォルト値を使用します。<br>
```

段落を分ける場合:

```md
この関数は設定ファイルを読み込みます。<br>

存在しない場合は、デフォルト値を使用します。<br>
```
