# Review Git Changes Three Times

`HEAD` に対する現在の作業ツリーの変更をレビューし、根拠が明確で直す価値のある問題を修正する。
実装によって関連ドキュメントが古くなる場合は更新し、この工程を正確に3回行う。

## 対象を確定する

1. 適用される `AGENTS.md` などの指示と、プロジェクト固有の規約を読む。
2. `git status --short --untracked-files=all`、`git diff --cached`、`git diff`、`git diff HEAD` を確認する。
3. tracked な staged / unstaged の変更に加え、`git status` に出る untracked ファイルも対象に含める。
4. 変更ファイルの全体と、差分の意図や影響を判断するために必要な呼び出し元、テスト、設定、ドキュメントを読む。
5. 利用可能なテスト、lint、型チェック、ビルドなどの検証手順を把握する。

ユーザーが対象を限定している場合は、その範囲だけをレビュー・修正し、範囲外の変更には触れない。
レビュー対象の変更がなければ、3回のループは行わず、その旨を報告して終了する。

## 3回のレビュー・修正ループ

次の工程を正確に3回繰り返す。
問題や修正がない回も1回として数え、早期終了も4回目の追加レビューも行わない。

1. 最新の `git status --short --untracked-files=all`、`git diff --cached`、`git diff`、`git diff HEAD` を取得し直し、status に出る untracked ファイルの内容も読み直す。
2. 修正済みの内容を含む現在の差分全体をレビューする。
3. 根拠が明確で、今回の変更に直接関係する問題だけを修正する。
4. 必要なテストを追加・更新し、関連ドキュメントの要否を判断して必要なものだけ更新する。
5. 変更に見合う検証を実行する。今回の変更が原因の失敗は修正し、同じ回の中で再実行する。
6. 主な指摘、修正、ドキュメント更新、検証結果をその回の記録として残す。該当なしの場合も記録する。

各回では全体を確認したうえで、次の観点に重点を移す。

1. 1回目: 要件との整合、正しさ、回帰、エラー処理、セキュリティ、データ損失
2. 2回目: 境界条件、統合、型・API・データの契約、テストの十分性、ドキュメントとの整合
3. 3回目: 残存する不具合、可読性・保守性、不要な複雑さ、意図しない変更、最終差分の一貫性

検証失敗の修正と再実行は追加のレビュー回として数えない。
3回目の修正後に最終検証を行ってよいが、それを4回目のレビューにしない。

## 修正する問題を選ぶ

次のような、再現可能またはコードから具体的に確認できる問題を修正する。

- 不具合、回帰、セキュリティ問題、データ損失の危険
- 要件、型、API、設定、永続化形式などの契約違反
- 今回の変更に必要なテストの欠落や、変更と矛盾するテスト
- 誤解や将来の不具合を招く明確な保守性の問題
- 実装変更によって不正確になったドキュメントやコードコメント

次の変更は行わない。

- 主観的な好みだけに基づく書き換え
- 今回の差分と無関係な既存問題の修正
- 目的外の大規模リファクタや、根拠のない機能追加
- 曖昧な仕様やプロダクト判断を推測で確定する変更

重要だが安全に修正できない問題は、未解決事項として報告する。

## ドキュメントを更新する

外部挙動、API、設定、環境変数、セットアップ、コマンド、データ移行、運用手順、利用例が変わり、既存の説明が古くなる場合に関連ドキュメントを更新する。
公開契約が変わる場合は、対応するコードコメントや API ドキュメントも確認する。
内部実装だけが変わり、既存文書が引き続き正しい場合は更新しない。
生成された文書やファイルは直接編集せず、プロジェクトの生成元と正規の生成手順を使う。

## 検証する

- 各回で、変更箇所に対応する targeted test、lint、型チェック、ビルドなどを可能な範囲で実行する。
- 3回目の修正後に `git diff --cached --check` と `git diff --check` を実行し、untracked ファイルも空白エラーを確認したうえで、関連する最終検証を実行する。
- 実行できない検証は理由を記録する。
- 失敗が今回の変更によるものか既存のものかを区別し、根拠なく成功扱いしない。

## 変更を保護する

- 同じファイル内の無関係な hunk を含め、対象範囲内の問題修正に必要な箇所以外の既存変更を保持する。
- `git reset`、`git checkout`、`git clean`、`git stash` を使わない。
- ユーザーの明示的な依頼なしに、stage、commit、push、PR 作成を行わない。
- 今回の差分と直接関係しないファイルを変更しない。
- 変更の途中で見つけた無関係な問題は、必要なら報告に留める。

## 最終報告

次を簡潔に報告する。

- 1回目から3回目までの主な指摘、修正、検証結果
- 最終的な変更概要
- ドキュメントの更新内容、または更新不要と判断した理由
- 実行した検証と結果
- 未解決事項や、実行できなかった検証

## 完了条件

- 差分がある場合、レビュー、必要な修正、検証を正確に3回実施している。
- 各回で最新の差分全体を確認している。
- 明確な問題を修正済み、または修正しなかった理由を報告している。
- 関連ドキュメントの要否を確認している。
- ユーザーの既存変更を失っていない。
