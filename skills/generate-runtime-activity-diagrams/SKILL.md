# 実行時 Activity Diagram の生成

## 入力と前提

- 要件定義書と実装計画書の **2 ファイル**を受け取る。
- Markdown（`.md`）と AsciiDoc（`.adoc`）を受け付け、形式の混在も許可する。
- どちらかが不足する、文書の種別を判別できない、または読み取れない場合は生成せず、理由と不足情報を報告する。
- 入力文書は読み取り専用とし、ID・コメント・整形を含めて変更しない。
- 対象フローが指定されていなければ、文書から判別できるすべての論理フローを対象にする。指定されていれば、その範囲だけを対象にする。
- 2つの入力文書が同じディレクトリにある場合は、そのディレクトリへ出力する。別ディレクトリにある場合は、生成前に保存先を確認し、回答を得るまで書き込まない。

## 生成前の全体チェック

1. 2文書を読み、入口、終了点、処理主体、状態変更、外部I/O、分岐、例外、ループ、並列処理を抽出する。
2. 要件定義書を期待動作の正本として、実装計画書を処理主体・連携先・技術的詳細の補助資料として照合する。
3. 処理順、分岐条件、終了条件などを変える重大な矛盾をすべてのフローについて先に検査する。
4. 重大な矛盾が **1件でも** あれば、全フローのファイルを書き込まず、矛盾箇所と確認事項を報告する。
5. 主要フローを変えない軽微な不足や曖昧さは、推測で確定せず `要確認` の note として記録し、生成を続ける。
6. 文書から論理フローを判別できない場合は、推測で図を作らず、対象範囲の確認を求める。

## 実行時フローの抽出

- 実装タスクの作業順ではなく、完成した機能が実行されるときの入力から終了までを描く。
- 1つの独立した入口・目的・終了点を持つ処理を1論理フローとして扱い、1論理フローにつき1組の `.puml` と `.png` を作成する。
- 正常系に加え、文書に明記された次の経路を含める。
  - 条件分岐と代替経路
  - エラー、例外、早期終了
  - リトライとループ
  - 並列処理と合流
- コード行、単純な実装タスク、文書にない関数・クラス・構成要素は追加しない。関数名やクラス名は入力文書に明記され、レビュー上有用な場合だけ使う。
- 利用者、フロントエンド、バックエンド、DB、外部サービスなど、複数の処理主体が文書に明記されている場合は主体ごとのスイムレーンを使う。文書にない主体を推測しない。
- アクティビティ名、条件名、処理主体名は入力文書の言語と用語を維持し、翻訳や不要な言い換えをしない。

## 出典と曖昧さ

- 各アクティビティ、分岐、終了条件、主要な状態変更の直前に、根拠となる `パス + 見出し` または `パス + 既存ID` を PlantUML コメントで記録する。
- 複数箇所を根拠にした要素には、すべての出典をコメントする。可能ならリポジトリ相対パスを使い、絶対パスは避ける。
- 出典コメントはレンダリング結果に表示しない。
- 軽微な不足・曖昧さは該当要素の近くに次の形式の可視 note を置く。

  ```plantuml
  note right
    要確認: 未確定事項を入力文書の用語で記述
  end note
  ```

各ファイルの先頭には、少なくとも次の生成情報をコメントで記録する。

```plantuml
' generated-by: generate-runtime-activity-diagrams
' requirements: path/to/requirements.md
' implementation-plan: path/to/implementation-plan.adoc
' flow: 論理フロー名
```

## PlantUML の作成

- [PlantUML Activity Diagram (beta)](https://plantuml.com/ja/activity-diagram-beta) の構文だけを使う。旧式 Activity Diagram 構文と混在させない。
- 必要に応じて `start`、`stop`、`if`／`elseif`／`else`／`endif`、`while`／`endwhile`、`repeat`／`repeat while`、`fork`／`fork again`／`end fork` を使う。
- 主体を表す場合は `|処理主体|` のスイムレーン構文を使う。
- 文書にない分岐、正常化、暗黙のリトライ、暗黙の並列化を補わない。
- ラベルに `:` や `;` など PlantUML の構文文字が含まれる場合は、意味を変えずに安全にエスケープまたは分割して構文を壊さない。
- 生成後、`@startuml` と `@enduml`、分岐・ループ・並列構文の対応を確認する。

## ファイル名と上書き

- 実装計画書の拡張子を除いた名前と論理フロー名から、次の同じベース名を作る。
  - `<実装計画書名>-<flow-name>-activity.puml`
  - `<実装計画書名>-<flow-name>-activity.png`
- 明示されたフローIDまたは見出し名を `flow-name` の基礎にする。空白をハイフンへ置換し、`/`、`\\`、制御文字、OSの予約文字をハイフンへ置換し、連続ハイフンを畳む。意味のない翻訳や新しい識別子の発明はしない。
- 生成予定の全ファイルについて、書き込み前に存在と出所を確認する。
- `.puml` の生成マーカーと2つの入力元が今回と一致し、対応する `.png` も同じベース名である場合だけ、確認なしで再生成する。
- マーカーがない、入力元が異なる、手書き・出所不明、孤立した既存PNGの場合は上書きせず、確認を求める。
- 1件でも上書き確認が必要なら、全フローの書き込みを止めてからまとめて確認する。

## PNG のローカル生成

- `plantuml` コマンド、またはユーザーが指定したローカル PlantUML jar と `java` を確認する。
- 公開 PlantUML サーバーやその他の外部レンダリングサービスへ、入力文書や `.puml` を送信しない。
- `plantuml -tpng <file.puml>`、または指定された jar に対する `java -jar <plantuml.jar> -tpng <file.puml>` を使い、各 `.puml` を同じディレクトリへ変換する。PNGの生成成功を構文検証の証拠として扱う。
- 複数ファイルを処理する場合も、入力パスを明示してローカルコマンドへ渡す。入力文書をアップロードするサービスや、ダウンロードを伴う自動フォールバックは使わない。
- ローカル実行環境がない場合は `.puml` だけを保存し、「PNG生成未完了」と明示する。依存関係の導入を勝手に行わず、ユーザーへ確認する。
- レンダリングに失敗した場合は `.puml` を残し、エラーとPNG未生成を報告する。PNGが1つでもない場合は成功扱いにしない。

## 完了条件と報告

次のすべてを満たした場合だけ、処理を完了とする。

- 対象フローごとに `.puml` と `.png` が存在し、PNGが現在の `.puml` からローカル生成されている。
- 正常系と、文書に明記された分岐、例外、終了、リトライ、ループ、並列処理が反映されている。
- 明記された複数の処理主体にはスイムレーンがあり、各主要要素に出典コメントがある。
- 重大な矛盾が残っておらず、軽微な曖昧さは `要確認` note として表示されている。
- 入力文書が変更されていない。

最後に、生成した全ファイルのパス、対象フロー、`要確認` の一覧、未完了事項、PNGの検証結果を報告する。PNGが未生成なら、処理未完了であることと依存関係導入の確認が必要なことを明示する。
